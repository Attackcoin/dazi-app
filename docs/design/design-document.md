# 搭子 App — 设计与实施计划
**日期：** 2026-03-08
**技术栈：** FlutterFlow + Firebase
**目标用户：** 非技术背景独立创始人

---

## Context

用户希望构建一款"搭子"社交 App，核心功能是：线上发布搭子需求 → 匹配感兴趣的人 → 线下见面。

**产品定位：反孤独引擎**
不是"找搭子工具"，而是用 AI 降低走出门的心理门槛，让孤独的人真正交到真朋友。

**时代背景（2026全球数据支撑）：**
- 54% 大学生每天感到孤独（美国6万人研究）
- Gen Z 70% 大量时间独处，1/3 主动删掉社交App
- Google搜索"跑步俱乐部"5年增长3倍 — 线下社交需求爆发
- Tinder -3%，Hinge +26% — 游戏化刷脸失败，真实连接崛起
- 没有一款AI社交App突破千万DAU — AI alone 解决不了孤独

**核心卡位：AI降低门槛 + 线下真实连接，两者同时做到的产品目前不存在。**

与竞品差异化：防鸽子机制（签到+爽约记录）+ 结构化发布（男女比例+费用方式）+ AI全流程辅助。冷启动免费，后期交易抽佣。

**发布策略：两阶段**
- 阶段一（当前）：国内版，微信/支付宝支付，阿里云实名认证，国内App市场上架
- 阶段二（产品验证后）：出海版，注册香港公司，Stripe支付，Persona KYC，目标台湾/新加坡/日本

## 技术架构

```
Frontend:  FlutterFlow（可视化构建，发布 iOS + Android）
Backend:   Firebase
  ├── Authentication  → 手机号 / Google / Apple 三种登录
  ├── Firestore       → 主数据库
  ├── Cloud Functions → 防鸽子逻辑 / AI解析 / 未来抽佣逻辑
  ├── FCM             → 推送通知
  └── Storage         → 头像 / 图片
Location:  Google Maps API（GPS 自动定位 + 城市手动切换）
AI:        Claude Haiku API（语音发布结构化解析）
Voice:     设备原生 Speech-to-Text（iOS/Android 内置，免费）
身份验证:  阿里云实人认证（身份证OCR + 活体检测 + 人脸比对）
支付:      微信支付 + 支付宝（担保交易，押金托管）+ Apple Pay
信用:      支付宝芝麻信用 API（高分用户免押金）
```

### 语音 + AI 快速发布

用户点击麦克风按钮，说一句话，AI 自动填好表单：

```
用户说：「周六下午三点，找两个人一起打羽毛球，在静安体育馆，AA制」
         │
         ▼
Google Speech-to-Text → 转为文字
         │
         ▼
Claude API (Haiku) → 解析结构化字段：
  {
    category: "运动 > 羽毛球",
    time: "周六 15:00",
    location: "静安体育馆",
    totalSlots: 3,  // 发布者+2人
    costType: "aa",
    title: "周六羽毛球搭子",
  }
         │
         ▼
自动填入表单，用户检查后点「发布」
```

**技术实现：**
- 语音转文字：设备原生 API（iOS/Android 均内置，免费）
- AI 解析：Claude Haiku API（每次调用约 ¥0.003，极低成本）
- 解析 Prompt 在 Firebase Function 中管理，方便后期优化

### 防鸽子机制
```
双方确认见面
    │
    ▼
见面当天解锁「签到」按钮（提前30分钟开放）
    │
    ▼
签到方式（二选一）：
  A. 扫对方专属二维码
  B. 同地点 GPS 验证（误差<100m）
    │
    ├── 双方签到成功 → 解锁「互评」功能
    │
    └── 超时 30 分钟未签到
            │
            ▼
        爽约方 ghostCount +1（公开显示在主页）
        注意：发布者爽约同样计入
        双方都不签到 → 双方各 +1
        ghostCount >= 3 → isRestricted = true（无法发布新搭子）
```

## 信任与安全体系

### 真人验证 — 三级体系
```
Level 1（注册，所有用户）
  手机号 OTP 验证 → 已有 ✅

Level 2（第一次参与搭子）
  自拍活体检测（眨眼/转头）→ 证明是真人
  服务商：腾讯云人脸核身 / 阿里云实人认证
  成本：约 ¥0.5/次

Level 3（发起搭子 + 开通押金）
  身份证 OCR + 人脸比对实名认证
  与公安数据库实时比对，准确率 99%+
  服务商：阿里云金融级实人认证
  成本：约 ¥1-2/次
```
核心逻辑：发布者比参与者承担更多责任，必须实名才能发帖。

### 押金系统（防鸽子经济机制）
**法律合规路径：** 微信支付「担保交易」，钱在微信侧托管，App不直接持有资金（避免"二清"违规）。

需提前准备：**营业执照 + 微信支付商户号 + 支付宝商家账号**

### 支付渠道（多选）
| 渠道 | 用途 | 说明 |
|------|------|------|
| 微信支付 | 押金担保 + 日常收付 | 最主流，覆盖最广 |
| 支付宝 | 押金担保 + 日常收付 | 备用渠道，支持担保交易 |
| Apple Pay | iOS 快捷支付 | 苹果用户体验更流畅 |
| 数字人民币（e-CNY）| 未来可选 | 政策支持，二阶段考虑 |

### 芝麻信用（支付宝）— 信用替代押金机制
芝麻信用分高的用户，押金可减免或直接豁免，降低参与门槛同时保持信任：

```
芝麻信用 750+ → 押金全免，信用担保代替
芝麻信用 650-750 → 押金减半（¥5-15）
芝麻信用 650以下 / 未授权 → 正常押金（¥10-30）
```

个人主页公开显示：「芝麻信用担保用户 ✓」标签，提升可信度。
接入方式：支付宝开放平台「芝麻信用」API（需商家资质）。

```
用户确认参加 → 自动查询芝麻信用分
                ├── 750+ → 免押金，信用背书
                └── 750以下 → 微信/支付宝担保支付冻结押金
                         ↓
             见面当天双方签到
                ├── 成功签到 → 自动解冻退还
                └── 一方爽约（超时30分钟）
                         ↓
                  扣款：爽约方押金 / 信用扣分
                  分配：出席者平分 90% + 平台 10%
```

**取消退款规则：**
| 取消时间 | 退款比例 |
|---------|---------|
| 见面前 24h+ 取消 | 全额退还 |
| 见面前 2-24h 取消 | 退还 50% |
| 当天取消 / 直接不到 | 0% 退还 |
| 发布者解散活动 | 所有人全额退还 |

**完整信任飞轮：**
```
实名认证 + 押金冻结 + 爽约记录（公开）+ 发布者勋章
= 国内目前最完善的搭子信任体系
```

## 趋势级功能（基于2026全球社交大趋势）

### 1. AI破冰助手（解决最大痛点：社交焦虑）
见面前24小时，AI分析双方资料，各自推送3条个性化破冰话题：
```
「你们都喜欢爬山，可以聊聊各自最难忘的一次经历」
「他/她最近在学日语，你可以问问学习契机」
「你们都是第一次来这家餐厅，可以一起研究菜单」
```
实现方式：Claude Haiku 读取双方标签+历史搭子记录生成，每次约¥0.01。

### 2. 胆小鬼模式（服务最大未被满足的群体：社恐用户）
```
用户设置「社恐保护模式」→ 广场自动过滤：
  ✅ 显示：3人以上小组 / 有结构化活动（打球/桌游/观影）
  ❌ 隐藏：纯吃饭聊天局 / 1对1 / 20人以上大型局
```
发布者可标记活动为「社恐友好」，获得额外流量加权推荐。

### 3. 即刻出发（Real-time 搭子，抓住临时社交冲动）
区别于提前预约，「即刻出发」专为临时起意设计：
```
我现在在三里屯 → 发起「30分钟内见面」邀请
→ 附近用户收到推送
→ 15分钟内确认人数达到 → 自动成团
→ 超时未成团 → 自动取消，无爽约记录
```
触发场景：下班后、周末无聊、旅行异地临时找搭子。

### 4. 社交成长月报（AI生成，制造留存+朋友圈裂变）
每月1日自动生成，格式参考网易云年度报告：
```
┌──────────────────────────────────┐
│    🎉 你的2月搭子报告             │
│                                   │
│    🤝 新认识了 8 位朋友            │
│    📍 探索了 5 个新地方            │
│    🏆 最佳搭子：小林（已见3次）    │
│    ⏰ 你的黄金时段：周六晚8点      │
│    🌟 你的搭子标签：美食探索家     │
│                                   │
│    「你比上个月更勇敢了一点」       │  ← AI个性化文案
│    [分享到朋友圈]                  │
└──────────────────────────────────┘
```
分享带App水印 → 新用户扫码下载 → 免费裂变增长。

### 5. 连接飞轮设计（从搭子→真朋友的路径）
```
首次搭子
    ↓
见面后互相「添加为搭子好友」
    ↓
系统提示：「你们已经见面2次了，要不要建个固定搭子群？」
    ↓
固定搭子群：同一批人定期聚，系统帮你发起下一次
    ↓
用户形成真实友谊 → 无法离开App → 自然留存
```
这是区别于所有现有社交App的核心：**不只是让陌生人见面一次，而是帮人建立真实的长期友谊。**

## 开发者管理指南

### 三套环境隔离（绝对不能混用）
```
Development（开发测试）
  └── Firebase项目：dazi-dev
  └── 随意测试，数据可清空，真实API可用假Key

Staging（预发布）
  └── Firebase项目：dazi-staging
  └── 上线前完整测试，数据结构与生产一致
  └── 用于 TestFlight / 内测APK

Production（生产）
  └── Firebase项目：dazi-prod
  └── 真实用户数据，严禁直接修改
  └── 所有变更必须经过 Dev → Staging → Prod 流程
```

### API 密钥安全管理
```
绝对不能做：
  ❌ 把 API Key 写死在 FlutterFlow 代码里
  ❌ 把密钥上传到任何公开代码仓库

正确做法：
  ✅ 所有第三方 API（Claude/阿里云/微信支付）密钥
     → 存储在 Firebase Functions 环境变量里
     → FlutterFlow 调用自己的 Firebase Function
     → Function 再调用第三方 API（密钥不暴露给前端）

  ✅ 用 Firebase Remote Config 管理可动态修改的配置
     （如：押金默认金额、各功能开关）
```

### FlutterFlow 版本管理
```
FlutterFlow 内置版本历史（类似 Git commit）：
  → 每完成一个功能模块，手动创建「版本快照」
  → 快照命名规范：v0.1-登录注册 / v0.2-广场首页 / v1.0-正式发布
  → 出问题可一键回滚到任意快照

导出代码策略：
  → 定期导出 Flutter 源码备份到本地
  → 复杂自定义功能（支付/实名认证）在导出代码里写，不在FlutterFlow拖拽
```

### Firebase Functions 部署规范
```
每次改动 Functions：
  Step 1: 在 dev 环境测试通过
  Step 2: 部署到 staging，完整测试押金/签到逻辑
  Step 3: 确认无误，部署到 prod
  Step 4: 观察 Firebase Console 错误日志30分钟

命令：
  firebase deploy --only functions --project dazi-prod
```

### 监控与告警体系
```
Firebase Crashlytics（崩溃监控）：
  → App崩溃自动上报，邮件通知
  → 每周查看崩溃率，目标 < 0.1%

Firebase Performance（性能监控）：
  → 监控页面加载时间、网络请求耗时
  → 广场首页加载目标 < 2秒

Firebase Analytics 关键看板（每周必看）：
  → DAU/MAU 趋势
  → 注册漏斗（各步骤流失率）
  → 核心事件：post_created / application_sent / checkin_success
  → 7日留存率（目标 > 30%）

设置告警：
  → Crashlytics：崩溃率 > 1% → 立即邮件通知
  → Functions 错误率 > 5% → 立即通知
```

### App 版本发布流程
```
每次发版（建议双周一次）：
  Week 1: 开发新功能
  Week 2: 测试 + 修Bug
  Day 12: 提交 TestFlight 内测（iOS）/ 发 APK 内测（Android）
  Day 14: 收集内测反馈，修复关键问题
  Day 15: 提交 App Store + Google Play 正式审核
  Day 17-18: 审核通过，正式发布

版本号规范：
  v1.0.0 → 首次正式发布
  v1.1.0 → 新功能（如即刻出发上线）
  v1.0.1 → Bug修复
  v2.0.0 → 重大改版（如出海国际版）
```

### Firestore 数据结构变更规范
```
数据库字段变更是高风险操作：
  ❌ 不能直接删除生产环境字段（会导致旧版App崩溃）

正确步骤：
  Step 1: 新版本App同时支持新旧字段（兼容期）
  Step 2: 观察老版本用户升级率（目标 > 90%）
  Step 3: 确认绝大多数用户已升级，才移除旧字段
  Step 4: 写 Migration Script 批量更新旧数据
```

灰度发布：新功能先对5%用户开启 → 观察数据 → 逐步扩大

| Firebase 费用过高 | 月费超过 ¥5000 | 评估迁移或优化查询 |
| FlutterFlow 无法实现需求 | 有功能拖拽做不了 | 导出代码转纯 Flutter |
| 查询性能下降 | 广场列表加载 > 3秒 | 加 Firestore 索引或引入缓存层 |
| 用户破10万 | DAU 稳定 > 1万 | 评估整体架构扩展性 |

### 最可能发生的升级：Firebase → 腾讯云 CloudBase

结构几乎一一对应，迁移成本可控：

```
Firebase Auth       →  腾讯云登录服务
Firestore           →  CloudBase 数据库（同为文档型）
Firebase Storage    →  腾讯云 COS 对象存储
Firebase Functions  →  腾讯云 SCF 云函数
Firebase FCM        →  腾讯云 TPNS 推送
Firebase Analytics  →  腾讯云 MTA 移动分析
```

**迁移步骤（渐进式，不停服）：**
```
Step 1: 新用户写入两套数据库（双写期）
Step 2: 验证两套数据一致性
Step 3: 读取切换到新数据库
Step 4: 停止双写，完全迁移
Step 5: 老数据库只读保留30天后关闭

关键原则：绝不做停服大迁移，用户无感知
```

### FlutterFlow → 纯 Flutter 迁移

当 FlutterFlow 成为瓶颈时（通常在有复杂自定义需求时）：

```
FlutterFlow 支持「导出完整 Flutter 代码」：
  → 导出后用 VS Code / Android Studio 继续开发
  → 已有页面和逻辑完整保留
  → 之后不再用 FlutterFlow，纯代码维护

迁移时机：
  ✅ 需要复杂动画
  ✅ 需要硬件功能（蓝牙/NFC）
  ✅ FlutterFlow 拖拽实现某功能超过2天
  ❌ 不要因为"想要更多控制"就迁移（YAGNI原则）
```

### 数据库架构演进原则

```
规则1：只加字段，不删字段（等旧版本用户全部升级后再删）
规则2：新字段给默认值（避免旧数据读取报错）
规则3：重大结构变更写 Migration Script，用 Firebase Function 批量执行
规则4：迁移前先在 staging 跑，确认无误再 prod
规则5：记录每次 schema 变更（在 README 或文档里）

示例 Migration Script：
  // 给所有 users 加新字段 verificationLevel = 1（默认值）
  const users = await db.collection('users').get()
  users.forEach(doc => {
    if (!doc.data().verificationLevel) {
      doc.ref.update({ verificationLevel: 1 })
    }
  })
```

## 验证方式

1. **完整流程验证：** 两个测试账号走完：注册 → 实名认证 → 发布 → 分享 → 申请 → 确认 → 押金冻结 → 签到 → 评价 → 押金解冻
2. **防鸽子验证：** 模拟一方超时不签到，确认 ghostCount+1，押金正确扣除，3次后限制发布
3. **男女比例验证：** 发布「男2女2」局，确认第3个同性别申请被系统拦截
4. **申请过期验证：** 发起申请后不操作，24h后确认自动过期
5. **genderQuota校验：** 尝试设置男3女3但总人数4，确认系统拦截报错
6. **分享验证：** 分享到微信/朋友圈，未安装用户点击链接能看到H5落地页，安装后DeepLink正确跳转
7. **未成年人验证：** 填写未成年出生年份，确认押金功能被禁用
8. **空状态验证：** 切换到无搭子城市，确认引导发布页正常显示
9. **上架验证：** TestFlight 内测通过后再提交正式审核
